Ktor Discord API Help

Se connecter à la Gateway

login(intents) ouvre la connexion WebSocket à la Gateway Discord. À partir de là, le bot reçoit les événements et peut mettre à jour sa présence. L'appel synchronise aussi les commandes déclarées et renvoie le Job de la boucle de connexion.

// Ouvre la connexion Gateway. Renvoie le Job de la boucle de connexion. val job = kda.login(intents = 1 or 512 or 32768) // runBlocking { } garde le processus vivant tant que le bot tourne.
intents

Le bitfield des intents Gateway (voir ci-dessous).

Enregistre tous tes handlers (on<T>, on(name) { }) avant login() — la boucle de dispatch démarre à la création du client, mais tu ne veux manquer aucun événement précoce.

Les intents

Les intents disent à Discord quelles familles d'événements tu veux recevoir. C'est un bitfield: on combine les bits voulus avec l'opérateur or.

Dans cette version, login prend un Int brut. Il n'existe pas encore d'enum typé pour les intents : tu passes directement la valeur du bit (ou la combinaison). Les valeurs ci-dessous sont celles de l' API Discord.

// Les intents sont un bitfield : on combine les bits voulus avec `or`. val GUILDS = 1 shl 0 // 1 val GUILD_MESSAGES = 1 shl 9 // 512 val MESSAGE_CONTENT = 1 shl 15 // 32768 kda.login(intents = GUILDS or GUILD_MESSAGES or MESSAGE_CONTENT)

* = intent privilégié: il doit être coché dans le portail développeur, sinon la connexion échoue.

Sans l'intent MESSAGE_CONTENT, le champ content des MessageCreateEvent arrive vide pour les messages qui ne mentionnent pas le bot. C'est la cause n°1 de « mon bot ne voit pas les messages ».

Cycle de vie de la connexion

login gère seul la robustesse de la connexion : heartbeat, détection de connexion « zombie », reconnexion avec backoff et resume de session. Tu n'as rien à câbler. Le détail est décrit dans Cycle de vie de la Gateway.

Pour fermer la connexion (et tout le reste), voir close().
